# Activity 3 Design a Data Protocol

![Data protocol](../assets/week-03/data%20protocol.jpg)

I designed a system to measure the volume of sounds in the surrounding environment. The frequency is to collect data every 10 seconds, and the mapping ranges from quiet (small squares), medium (medium squares), to loud (large squares). If there is no sound for more than 10 seconds, a line is drawn.

After revising my data protocol, some parts intrigued me. For example, I didn't specifically configure whether the next record should be placed next to the previous one, but this seemed to be based on habitual recording, and the volume of some recorded times maintained a rhythm from low to high, except for a few particularly loud nodes that surprised me.

This process allowed me to understand the malleability of data truly: day and night and cloud cover control the background color, not just visually, but also conveying a sense of time and weather mood; temperature corresponds to the size and number of sides of the graphics, and air pressure corresponds to vibration, transforming abstract numbers into a more intuitive energy state; wind speed and rainfall control the movement of the particle system, creating a visual rhythm of alternating speeds. The entire process is like translating data into a living system, allowing viewers to feel the state and mood of Tokyo's weather without needing to understand the numbers.

During this process, I also explored more ways to collaborate with the AI: first, providing design intent, mapping strategies, and visualization rules; then, using the AI ​​to generate code; and finally adjusting and validating it myself. This workflow not only accelerated the implementation but also gave me more room to think about "how data is perceived" rather than just being presented. # Reflection

This week's content was incredibly rich and engaging, giving me a deeper understanding of the concept that "data is not just numbers, but a material that can be designed and translated." From creating animations with p5.js to integrating real-time weather data into the screen using an API, I began to realize that data itself has no emotion. Still, through design (such as visuals, motion, and interaction), it can be transformed into something with feeling. Using `noise()` to make data "move" was particularly inspiring; the process of transforming data into a living system was very enlightening.

Furthermore, in the data protocol for Activity 3, I discovered that "observation" is more interesting than the rules themselves. What started as a simple, rational setting of sound volume → block size revealed rhythms, sudden spikes, and even the influence of context (class dismissal, crowds)—contexts that are invisible to pure data. This made me think that in future data-related designs, I could also incorporate "unpredictability" or "human behavior" into the design process, rather than just pursuing precision.

This week also helped me become more familiar with workflows (terminals, APIs, website setup). While issues like image paths and caching seem minor, they significantly impact the final presentation, giving me a better understanding of the complete workflow of a database digital project.

Extending to real-life applications, I started thinking about how to transform these tools into more meaningful designs. For example, in my current emotion-related design project, emotions can be viewed as live data, transformed into dynamic visuals through user behavior (text input, interaction frequency, dwell time), making abstract emotions "visible" and "feelable." Simultaneously, I can combine concepts like APIs and incorporate environmental data (such as weather and time) into the design, allowing the entire experience to change with the real world and forming a more vibrant interactive system.

Overall, the biggest takeaway this week was realizing that I'm not just "making things with data," but rather "designing how data is perceived." This will be very helpful for my future work in interactive experiences and even vibe coding.

The following code represents the technical implementation generated in response to the prompt above:

```javascript
let weather;
let url = "https://api.open-meteo.com/v1/forecast?latitude=35.683899&longitude=139.788694&current=temperature_2m,relative_humidity_2m,wind_speed_10m,rain,weather_code,cloud_cover,pressure_msl,surface_pressure,is_day&timezone=Asia%2FTokyo";

let particles = [];
let numParticles = 150;

function preload() {
  weather = loadJSON(url);
}

function setup() {
  createCanvas(windowWidth, windowHeight);
  for (let i = 0; i < numParticles; i++) {
    particles.push(new AtmosphericParticle());
  }
}

function draw() {
  if (!weather) return;

  let curr = weather.current;
  let temp = curr.temperature_2m;
  let wind = curr.wind_speed_10m;
  let humidity = curr.relative_humidity_2m;
  let isDay = curr.is_day;
  let clouds = curr.cloud_cover;
  let rain = curr.rain;
  let pressure = curr.surface_pressure;

  let dayColor = color(135, 206, 235);
  let nightColor = color(10, 20, 45);
  let cloudOvercast = color(80, 85, 95);
  
  let baseTone = isDay === 1 ? dayColor : nightColor;
  let finalBG = lerpColor(baseTone, cloudOvercast, clouds / 100);
  background(finalBG);

  push();
  translate(width / 2, height / 2);
  
  let jitter = map(pressure, 960, 1040, 5, 0.5);
  let pulse = sin(frameCount * (pressure/500)) * jitter;
  
  let radius = map(temp, 0, 40, 50, 250);
  let sides = floor(map(temp, 0, 40, 3, 12)); 
  
  let coreColor = lerpColor(color(0, 150, 255), color(255, 100, 0), map(temp, 5, 35, 0, 1));
  
  stroke(coreColor);
  strokeWeight(2);
  noFill();
  
  for (let i = 0; i < 5; i++) {
    rotate(frameCount * 0.01);
    drawPolygon(0, 0, radius + (i * 10) + pulse, sides);
  }
  pop();

  for (let p of particles) {
    p.update(wind, rain);
    p.display(rain);
  }

  displayUI(temp, wind, humidity, rain, clouds, pressure);
}

function drawPolygon(x, y, radius, npoints) {
  let angle = TWO_PI / npoints;
  beginShape();
  for (let a = 0; a < TWO_PI; a += angle) {
    let sx = x + cos(a) * radius;
    let sy = y + sin(a) * radius;
    vertex(sx, sy);
  }
  endShape(CLOSE);
}

class AtmosphericParticle {
  constructor() {
    this.pos = createVector(random(width), random(height));
    this.vel = createVector(random(-1, 1), random(-1, 1));
    this.acc = createVector(0, 0);
  }

  update(wind, rain) {
    if (rain > 0) {
      let gravity = map(rain, 0, 10, 2, 10);
      this.pos.y += gravity;
      this.pos.x += random(-0.5, 0.5);
    } else {
      let windForce = map(wind, 0, 50, 0.5, 15);
      this.pos.x += windForce;
      this.pos.y += sin(this.pos.x * 0.02) * 2;
    }

    if (this.pos.x > width) this.pos.x = 0;
    if (this.pos.y > height) this.pos.y = 0;
  }

  display(rain) {
    if (rain > 0) {
      stroke(150, 180, 255, 150);
      line(this.pos.x, this.pos.y, this.pos.x, this.pos.y + 12);
    } else {
      noStroke();
      fill(255, 180);
      ellipse(this.pos.x, this.pos.y, 3, 3);
    }
  }
}

function displayUI(t, w, h, r, c, p) {
  fill(0, 150);
  noStroke();
  rect(20, 20, 260, 140, 8);
  fill(255);
  textFont('monospace');
  textSize(13);
  text(`LOCATION: TOKYO, JP`, 35, 45);
  text(`TEMP: ${t}°C | HUMID: ${h}%`, 35, 70);
  text(`WIND: ${w}km/h | RAIN: ${r}mm`, 35, 90);
  text(`CLOUDS: ${c}%`, 35, 110);
  text(`PRESSURE: ${p}hPa`, 35, 130);
}
```
